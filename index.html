1. Vue d'ensemble — comment tout est relié

Le projet a deux programmes complètement séparés qui communiquent uniquement à travers la base de données MySQL. Ce n'est pas Django qui reçoit les données des capteurs : c'est un script Python indépendant (collecte.py) qui tourne en permanence sur un serveur et qui écrit dans la base. Django, lui, ne fait que lire (et un peu modifier) ce que ce script a déjà écrit.

Le lien entre les deux projets est uniquement la base de données MySQL. collecte.py fait des INSERT dans les tables capteur et mesure. Django (fichier models.py) déclare des classes Python qui correspondent exactement à ces deux mêmes tables (avec managed = False, ce qui dit à Django "je ne crée pas ces tables, elles existent déjà, je viens juste les lire/écrire"). Si demain on enlève collecte.py, le site Django continue de marcher mais ne reçoit plus de nouvelles données.

2. Collecte.py — le script qui récupère les données MQTT

Question piège probable du prof : "il y a une fonction asynchrone, comment ça marche ?"
En réalité il n'y a aucun async def / await dans ce code (pas de vraie programmation asynchrone Python). Ce qui se passe et qui ressemble à de l'asynchrone, c'est le modèle événementiel par callbacks de la librairie paho-mqtt : le programme ne demande pas activement "y a-t-il un message ?" en boucle. Il s'abonne une fois, puis lance client.loop_forever() qui bloque le programme et attend. Quand un message arrive sur le réseau, la librairie appelle automatiquement la fonction on_message — c'est ça la logique "asynchrone" / événementielle : le code ne s'exécute pas dans l'ordre du fichier, il s'exécute en réaction à un évènement réseau, à un moment imprévisible. Dis bien ça au prof : "ce n'est pas de l'asyncio, c'est un système de callbacks déclenchés par le réseau".

2.2 — parse_message() : transformer le texte brut en données utilisables

32-57
def parse_message(payload: str) -> dict: brut = {} derniere_cle = None for morceau in payload.split(","): if "=" in morceau: cle, valeur = morceau.split("=", 1) brut[cle] = valeur derniere_cle = cle elif derniere_cle is not None: brut[derniere_cle] = brut[derniere_cle] + "." + morceau
La trame envoyée par le capteur ressemble à : Id=12A6B8AF6CD3,piece=sejour,date=15/06/2026,heure=12:13:14,temp=26,35. Le problème : la température utilise une virgule comme séparateur décimal (26,35 au lieu de 26.35), alors que la trame entière est aussi séparée par des virgules ! Donc payload.split(",") va casser "26,35" en deux morceaux : "temp=26" et "35".

La boucle reconstruit ça intelligemment : pour chaque morceau, si on trouve un =, c'est une nouvelle clé (ex: temp → 26), et on retient cette clé dans derniere_cle. Si le morceau n'a pas de = (cas du "35" tout seul), ça veut dire que c'est la suite de la valeur précédente : on le recolle avec un point, ce qui transforme "26" en "26.35" — la bonne notation décimale Python/SQL.
49-57
heure = brut.get('heure') or brut.get('time') dt = datetime.strptime(f"{brut['date']} {heure}", "%d/%m/%Y %H:%M:%S") return { "id": brut["Id"], "piece": brut["piece"], "timestamp": dt.strftime("%Y-%m-%d %H:%M:%S"), "temperature": float(brut["temp"]), }
brut.get('heure') or brut.get('time') : gère le fait que selon le capteur, la clé peut s'appeler "heure" ou "time" (tolérance aux variantes). datetime.strptime(...) : convertit le texte "15/06/2026 12:13:14" (format français, jour/mois/année) en un véritable objet date Python. .strftime("%Y-%m-%d %H:%M:%S") : le reconvertit en texte mais cette fois au format MySQL standard (année-mois-jour), qui est le seul format que la colonne TIMESTAMP/DATETIME de MySQL accepte sans ambiguïté. Le dictionnaire final est "propre" et normalisé, prêt pour la base.

2.3 — Accès base de données : get_connexion() et inserer_mesure()

63-73
def get_connexion(): global _conn try: if _conn is None: _conn = pymysql.connect(**DB) else: _conn.ping(reconnect=True) return _conn except pymysql.MySQLError: _conn = None return None
Fonction de résilience réseau. global _conn permet de modifier la variable globale définie ligne 25. Si on n'a jamais de connexion (None), on en crée une nouvelle. Si on en a déjà une, .ping(reconnect=True) vérifie qu'elle est toujours vivante et la rouvre automatiquement si le serveur MySQL a coupé la connexion (timeout, redémarrage...). Si ça plante quand même (MySQLError), on remet _conn à None et on retourne None → le code appelant comprendra "la base est injoignable" et passera en mode cache.
76-92
def inserer_mesure(conn, m: dict): with conn.cursor() as cur: cur.execute("SELECT id FROM capteur WHERE id = %s", (m["id"],)) existe = cur.fetchone() if existe is None: cur.execute( "INSERT INTO capteur (id, nom, piece, emplacement) " "VALUES (%s, %s, %s, %s)", (m["id"], m["id"], m["piece"], m["piece"]), ) cur.execute( "INSERT INTO mesure (capteur_id, timestamp, temperature) " "VALUES (%s, %s, %s)", (m["id"], m["timestamp"], m["temperature"]), ) conn.commit()
C'est ici que les requêtes SQL brutes (pas d'ORM, contrairement à Django) sont exécutées. conn.cursor() ouvre un curseur (l'objet qui exécute des requêtes). Étape 1 : SELECT id FROM capteur WHERE id = %s vérifie si ce capteur existe déjà dans la table capteur (table "catalogue" des capteurs connus). %s est un paramètre préparé : PyMySQL remplace ça par la vraie valeur de façon sécurisée, ce qui protège contre les injections SQL (jamais de concaténation de chaîne directe). Étape 2 : si existe is None (capteur jamais vu), on l'INSERT dans capteur en utilisant l'ID comme nom et la pièce comme emplacement par défaut (valeurs provisoires, modifiables ensuite depuis Django). Étape 3 : dans tous les cas, on insère la nouvelle mesure dans la table mesure, reliée au capteur via la colonne capteur_id (clé étrangère). conn.commit() valide définitivement la transaction (sans ça, MySQL annulerait tout à la fermeture).

2.4 — Le cache local (anti-perte de données)

98-116
def mettre_en_cache(m: dict): with open(CACHE, "a", encoding="utf-8") as f: f.write(json.dumps(m) + "\n") def vider_cache(conn): if not os.path.exists(CACHE): return with open(CACHE, encoding="utf-8") as f: en_attente = [json.loads(ligne) for ligne in f if ligne.strip()] if not en_attente: return for m in en_attente: inserer_mesure(conn, m) os.remove(CACHE)
Mécanisme de tolérance aux pannes : si MySQL est temporairement injoignable (réseau coupé, serveur en maintenance), on ne perd pas la mesure. mettre_en_cache l'ajoute en mode append ("a") à un fichier texte, une ligne JSON par mesure (format JSON Lines, pratique car on peut ajouter sans relire tout le fichier). vider_cache fait l'inverse : à la reconnexion, elle relit toutes les lignes en attente, les réinsère une par une avec inserer_mesure, puis supprime le fichier cache une fois tout rejoué — pour ne pas les réinsérer deux fois la prochaine fois.

2.5 — Les callbacks MQTT (le coeur de la logique "événementielle")

122-129
def on_connect(client, userdata, flags, reason_code, properties): if reason_code == 0: for topic in TOPICS: client.subscribe(topic) else: print(f"Echec de connexion (code {reason_code})")
Cette fonction n'est jamais appelée directement par nous : on la "branche" plus loin (ligne 162) avec client.on_connect = on_connect. La librairie paho-mqtt l'appelle automatiquement dès que la connexion TCP au broker réussit. reason_code == 0 signifie "succès" (norme MQTT). Si succès, on s'abonne (subscribe) à chaque topic de la liste — c'est seulement après cet abonnement qu'on commencera à recevoir des messages.
132-154
def on_message(client, userdata, msg): payload = msg.payload.decode("utf-8", errors="ignore").strip() try: m = parse_message(payload) except (KeyError, ValueError) as e: print(f"Trame ignoree ({e}) : {payload}") return ... conn = get_connexion() if conn is not None: try: vider_cache(conn) inserer_mesure(conn, m) except pymysql.MySQLError as e: mettre_en_cache(m) else: mettre_en_cache(m)
C'est LA fonction "callback" qui se déclenche toute seule à chaque message MQTT reçu — c'est elle qui justifie le comportement "asynchrone". msg.payload arrive en bytes (octets bruts du réseau), donc .decode("utf-8") le convertit en texte lisible. On essaie de parser la trame ; si elle est mal formée (clé manquante = KeyError, ou date invalide = ValueError), on l'ignore proprement plutôt que de crasher tout le programme. Ensuite on tente la base : si elle répond (conn is not None), on vide d'abord le cache en attente (pour respecter l'ordre chronologique), puis on insère la mesure du jour. Si la base ne répond pas ou qu'une erreur survient pendant l'insertion, la mesure part au cache pour être rejouée plus tard. Aucune mesure n'est donc jamais perdue.

2.6 — main() : démarrage et boucle infinie

160-177
def main(): client = mqtt.Client(callback_api_version=mqtt.CallbackAPIVersion.VERSION2) client.on_connect = on_connect client.on_message = on_message client.reconnect_delay_set(min_delay=1, max_delay=30) client.connect(BROKER, PORT, keepalive=60) try: client.loop_forever() except KeyboardInterrupt: print("Arret demande par l'utilisateur.") finally: client.disconnect() if _conn is not None: _conn.close()
On crée l'objet client MQTT, puis on lui "branche" nos deux fonctions définies plus haut sur les évènements correspondants (on_connect / on_message) — c'est de l'inversion de contrôle : ce n'est pas nous qui appelons ces fonctions, c'est la librairie, quand elle juge que l'évènement s'est produit. reconnect_delay_set configure une reconnexion automatique avec un délai croissant (1s, puis jusqu'à 30s max) en cas de coupure réseau. client.connect(...) initie la connexion TCP. client.loop_forever() est une boucle bloquante : le programme s'arrête ici et attend indéfiniment, en laissant la librairie gérer la réception réseau et déclencher les callbacks en arrière-plan — c'est ce blocage volontaire + déclenchement différé qui donne l'impression d'asynchrone, même si techniquement c'est du code synchrone classique avec des callbacks. KeyboardInterrupt capture le Ctrl+C pour fermer proprement (déconnexion MQTT + fermeture de la connexion MySQL) au lieu de planter brutalement.

3. models.py — la correspondance Python ↔ tables MySQL (ORM)

Django utilise un ORM (Object-Relational Mapping) : chaque classe Python représente une table SQL, chaque attribut représente une colonne. On écrit du Python, Django génère le SQL tout seul.

4-15
class Capteur(models.Model): id = models.CharField(max_length=20, primary_key=True) nom = models.CharField(max_length=50, unique=True) piece = models.CharField(max_length=50) emplacement = models.CharField(max_length=100) class Meta: managed = False db_table = "capteur"
Représente la table capteur. id est une chaîne (l'adresse MAC du capteur, ex: "12A6B8AF6CD3") utilisée comme clé primaire (primary_key=True) — donc pas d'auto-incrément ici, c'est l'ID matériel du capteur qui sert d'identifiant unique. nom est unique=True : deux capteurs ne peuvent pas avoir le même nom (vérifié au niveau base de données). managed = False est crucial : ça dit à Django "ne touche jamais à la structure de cette table (pas de migration CREATE TABLE/ALTER TABLE), elle est déjà créée et gérée manuellement en SQL". db_table = "capteur" précise le nom exact de la table en base (sinon Django aurait cherché capteurs_capteur par convention).
18-31
class Mesure(models.Model): id = models.AutoField(primary_key=True) capteur = models.ForeignKey( Capteur, on_delete=models.CASCADE, db_column="capteur_id" ) timestamp = models.DateTimeField() temperature = models.DecimalField(max_digits=5, decimal_places=2) class Meta: managed = False db_table = "mesure"
Représente la table mesure (une ligne = une température mesurée à un instant donné). AutoField = clé primaire auto-incrémentée classique (1, 2, 3...). ForeignKey(Capteur, ...) crée le lien relationnel : chaque mesure appartient à un seul capteur, mais un capteur peut avoir plusieurs mesures (relation 1-à-plusieurs). db_column="capteur_id" précise que la colonne SQL réelle s'appelle capteur_id (convention Django par défaut de toute façon, mais explicité ici). on_delete=models.CASCADE : si on supprime un capteur, toutes ses mesures sont supprimées automatiquement par la base — c'est exactement ce qui se passe dans views.py ligne 109 (capteur.delete()) : Django/MySQL nettoie tout en cascade, on n'a rien à faire de plus. DecimalField(max_digits=5, decimal_places=2) : un nombre décimal précis à 2 chiffres après la virgule (ex: 26.35), avec maximum 5 chiffres au total — plus fiable qu'un float pour des valeurs scientifiques stockées en base (pas d'erreur d'arrondi binaire).
Pourquoi managed = False partout ? Parce que c'est collecte.py (et le SQL manuel du groupe) qui a créé les tables avant même que Django existe sur le projet. Django n'a donc jamais eu besoin de migration CREATE TABLE — d'où l'absence de fichiers de migration utiles dans capteurs/migrations/ (juste un __init__.py vide).

4. views.py — la logique métier (filtres, calculs, export)

4.1 — appliquer_filtres() : le moteur de recherche/filtre, partagé partout

12-21
def appliquer_filtres(request): mesures = Mesure.objects.select_related("capteur").all() capteur = request.GET.get("capteur", "").strip() if capteur: mesures = mesures.filter( Q(capteur__id__icontains=capteur) | Q(capteur__nom__icontains=capteur) | Q(capteur__piece__icontains=capteur) )
Mesure.objects.all() = équivalent ORM de SELECT * FROM mesure. .select_related("capteur") est une optimisation : sans ça, Django ferait 1 requête pour les mesures puis 1 requête par ligne pour récupérer le capteur correspondant (problème "N+1 requêtes"). Avec select_related, Django fait une seule requête avec un JOIN SQL — beaucoup plus rapide.

request.GET.get("capteur", "") récupère le paramètre d'URL ?capteur=... (ce qui a été tapé dans le champ de recherche). Q(...) permet de combiner plusieurs conditions avec un OU logique (|) — ici on cherche le texte dans l'ID, le nom, OU la pièce. icontains = "contient, insensible à la casse" → équivalent SQL LIKE '%valeur%' en ignorant majuscules/minuscules. Le double underscore capteur__id signifie "traverse la relation capteur, puis prends sa colonne id" (équivalent d'un JOIN + filtre sur la table jointe).
23-43
date_debut = request.GET.get("date_debut", "").strip() if date_debut: mesures = mesures.filter(timestamp__date__gte=date_debut) date_fin = request.GET.get("date_fin", "").strip() if date_fin: mesures = mesures.filter(timestamp__date__lte=date_fin) temp_min = request.GET.get("temp_min", "").strip() if temp_min: try: mesures = mesures.filter(temperature__gte=float(temp_min)) except ValueError: pass
Filtres de plage. timestamp__date__gte = "la partie date du timestamp est supérieure ou égale à" (équivalent SQL WHERE DATE(timestamp) >= '...'). __lte = inférieur ou égal. Même logique pour temperature__gte / __lte (température min/max). Le try/except ValueError: pass protège contre une entrée invalide (texte non numérique tapé dans le champ température) : au lieu de planter, le filtre est simplement ignoré silencieusement.
45-54
tris_valides = { "date": "timestamp", "temperature": "temperature", "capteur": "capteur__id", "nom": "capteur__nom", "piece": "capteur__piece", } tri = tris_valides.get(request.GET.get("tri", "date"), "timestamp") ordre = "" if request.GET.get("ordre", "desc") == "asc" else "-" return mesures.order_by(f"{ordre}{tri}")
Sécurité importante : on ne fait jamais directement order_by(request.GET.get("tri")), car ça permettrait à un utilisateur malveillant de trier sur n'importe quel champ caché ou de causer une erreur. À la place, on utilise un dictionnaire blanc-liste (tris_valides) : seules les 5 clés autorisées peuvent être utilisées, sinon ça retombe sur "timestamp" par défaut. Le préfixe "-" devant le nom de colonne en Django signifie "ordre décroissant" (le signe - est la syntaxe Django pour ORDER BY ... DESC), une chaîne vide donne l'ordre croissant (ASC).

4.2 — _points_graphe() : préparer les données pour Chart.js

57-67
def _points_graphe(mesures, max_points=120): points = list(mesures.order_by("timestamp")) n = len(points) if n > max_points: pas = n // max_points points = points[::pas][-max_points:] return ( [p.timestamp.strftime("%d/%m %H:%M") for p in points], [float(p.temperature) for p in points], )
S'il y a 5000 mesures, afficher 5000 points sur un graphique serait illisible et lent à dessiner. Cette fonction fait un sous-échantillonnage ("downsampling") : si on a plus de 120 points, on calcule un pas (par exemple si n=1200, pas=10) et on ne garde qu'1 point sur 10 (points[::pas], slicing Python avec un pas). [-max_points:] garantit qu'on ne dépasse jamais 120 points au final même après l'échantillonnage. La fonction renvoie un tuple de deux listes parallèles : les labels (dates formatées en texte) pour l'axe X, et les valeurs numériques pour l'axe Y — exactement ce que Chart.js attend en entrée.

4.3 — liste() : la page principale

73-98
def liste(request): mesures = appliquer_filtres(request) moyenne = mesures.aggregate(m=Avg("temperature"))["m"] labels, valeurs = _points_graphe(mesures) ... return render(request, "capteurs/liste.html", { "mesures": mesures[:500], "total_mesures": mesures.count(), "moyenne": moyenne, "capteurs": Capteur.objects.all(), "labels": json.dumps(labels), "valeurs": json.dumps(valeurs), ... })
C'est la vue branchée sur la page d'accueil (urls.py, path("", views.liste, name="liste")). On part toujours du même jeu filtré (appliquer_filtres), pour garantir la cohérence entre tableau, statistiques et graphique. .aggregate(m=Avg("temperature")) = équivalent SQL SELECT AVG(temperature) AS m FROM ... appliqué aux mêmes filtres (donc la moyenne affichée correspond bien à ce qui est filtré, pas à toute la base). mesures[:500] = limite Python/SQL à 500 lignes affichées (équivalent LIMIT 500) pour ne pas surcharger la page, alors que .count() donne le vrai total filtré (même s'il dépasse 500) pour informer l'utilisateur. render(...) assemble le template HTML avec ce contexte (dictionnaire de variables) — c'est ce dictionnaire qui devient accessible dans le template via {{ moyenne }}, {{ mesures }}, etc. json.dumps(labels) convertit la liste Python en texte JSON valide, injectable directement dans une balise <script> du template (voir liste.html ligne 103).

4.4 — detail() : fiche d'un capteur (lecture, édition, suppression)

104-120
def detail(request, capteur_id): capteur = get_object_or_404(Capteur, id=capteur_id) if request.method == "POST": if "supprimer" in request.POST: capteur.delete() return redirect("liste") nouveau_nom = request.POST.get("nom", "").strip() nouvel_emplacement = request.POST.get("emplacement", "").strip() if nouveau_nom: capteur.nom = nouveau_nom if nouvel_emplacement: capteur.emplacement = nouvel_emplacement capteur.save(update_fields=["nom", "emplacement"]) return redirect("detail", capteur_id=capteur.id)
capteur_id vient directement de l'URL (voir urls.py : path("capteur/<str:capteur_id>/", ...)). get_object_or_404 = équivalent SELECT * FROM capteur WHERE id = ... LIMIT 1, mais si rien n'est trouvé, Django renvoie automatiquement une page d'erreur 404 au lieu de planter.

Distinction GET / POST : un GET simple = on veut juste voir la page (pas de modification). Un POST = le formulaire HTML a été soumis (bouton "Enregistrer" ou "Supprimer" cliqué). "supprimer" in request.POST teste si le bouton qui a été cliqué portait l'attribut name="supprimer" (voir detail.html ligne 30) : si oui, capteur.delete() supprime la ligne dans capteur — et grâce au on_delete=CASCADE du modèle, MySQL supprime aussi automatiquement toutes les mesures liées. redirect("liste") renvoie ensuite vers la page d'accueil (pattern POST/Redirect/GET : on ne reste jamais sur une page après un POST, pour éviter qu'un rafraîchissement du navigateur ne resoumette le formulaire deux fois).

Si ce n'est pas une suppression, on met à jour le nom/emplacement seulement si le champ n'est pas vide (sinon on écraserait avec une valeur vide par erreur). save(update_fields=[...]) est une optimisation : ça génère un UPDATE SQL qui ne touche que ces deux colonnes, pas toutes (plus rapide, évite des conflits si une autre colonne a été modifiée entre-temps par collecte.py).
122-133
mesures = Mesure.objects.filter(capteur=capteur).order_by("-timestamp") moyenne = mesures.aggregate(m=Avg("temperature"))["m"] labels, valeurs = _points_graphe(mesures) return render(request, "capteurs/detail.html", {...})
Cette partie ne s'exécute que si la requête est un GET (affichage simple) — toutes les mesures de ce capteur précis, triées de la plus récente à la plus ancienne (-timestamp). Même logique de moyenne et de graphique que dans liste(), mais limitée à un seul capteur au lieu de tous.

4.5 — export_csv() : téléchargement des données filtrées

139-150
def export_csv(request): mesures = appliquer_filtres(request) reponse = HttpResponse(content_type="text/csv") reponse["Content-Disposition"] = 'attachment; filename="mesures.csv"' writer = csv.writer(reponse) writer.writerow(["capteur_id", "nom", "piece", "timestamp", "temperature"]) for m in mesures: writer.writerow([m.capteur_id, m.capteur.nom, m.capteur.piece, m.timestamp, m.temperature]) return reponse
Réutilise encore appliquer_filtres : l'export respecte les mêmes filtres actifs au moment du clic (voir le lien dans liste.html qui transmet les mêmes paramètres GET). HttpResponse(content_type="text/csv") crée une réponse HTTP qui n'est pas une page HTML mais un fichier. L'en-tête Content-Disposition: attachment; filename=... dit au navigateur "ne l'affiche pas, télécharge-le sous ce nom". csv.writer(reponse) écrit directement dans la réponse HTTP comme si c'était un fichier classique. On écrit d'abord la ligne d'en-tête (noms de colonnes), puis on boucle sur chaque mesure filtrée pour écrire une ligne CSV par mesure.

7. Templates HTML — l'affichage côté navigateur

7.1 — base.html : le squelette commun

1-15
{% load static %} <!DOCTYPE html> <html lang="fr"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> {% if refresh %}<meta http-equiv="refresh" content="{{ refresh }}">{% endif %} <script src="https://cdn.jsdelivr.net/npm/chart.js"></script> <link rel="stylesheet" href="{% static 'capteurs/style.css' %}"> </head> <body> {% block contenu %}{% endblock %} </body> </html>
{% load static %} active le système de gestion des fichiers statiques de Django (CSS/JS/images). Le meta viewport est ce qui rend le site adaptatif mobile : sans lui, les téléphones afficheraient le site comme une miniature de page bureau. {% if refresh %} : si l'utilisateur a réglé un rafraîchissement automatique (champ "Refresh auto (s)" dans le formulaire de filtres), on injecte une balise meta http-equiv="refresh" qui dit au navigateur de recharger toute la page après N secondes — utile pour suivre les températures en quasi temps réel sans JavaScript compliqué. Chart.js est chargé depuis un CDN externe (pas besoin de l'installer localement). {% block contenu %}{% endblock %} est un emplacement vide que les templates enfants (liste.html, detail.html) viennent remplir grâce à {% extends "capteurs/base.html" %} — c'est l'héritage de templates Django, qui évite de réécrire le head/body partout.

7.2 — liste.html : formulaire de filtres, statistiques, graphique, tableau

9-38
<form class="filtres" method="get"> <input type="text" name="capteur" value="{{ f_capteur }}"> <input type="date" name="date_debut" value="{{ f_debut }}"> ... <input type="hidden" name="tri" value="{{ tri }}"> <input type="hidden" name="ordre" value="{{ ordre }}"> <button type="submit">Filtrer</button> </form>
method="get" est volontaire (pas "post") : ça transforme chaque champ en paramètre d'URL (?capteur=...&date_debut=...), ce qui permet de partager/recharger/mettre en favori une recherche filtrée, et c'est exactement ce que request.GET.get(...) lit côté views.py. Les champs value="{{ f_capteur }}" réaffichent la valeur déjà tapée (pour que le formulaire ne se vide pas après soumission). Les deux type="hidden" conservent le tri actuel même quand on change un filtre, pour ne pas perdre l'état de tri en cours de route.
102-132
<script> const labels = {{ labels|safe }}; const valeurs = {{ valeurs|safe }}; new Chart(document.getElementById('graphe'), { type: 'line', data: { labels: labels, datasets: [{ data: valeurs, ... }] }, ... }); </script>
C'est ici que la donnée préparée côté Python (json.dumps(labels) dans views.py) devient un vrai tableau JavaScript directement dans le HTML. Le filtre |safe dit à Django "n'échappe pas ce texte" — normalement Django protège automatiquement contre le XSS en transformant les caractères spéciaux, mais ici on veut que les crochets [...] du JSON restent intacts pour être interprétés comme du JS valide (sans danger ici car c'est nous-mêmes qui générons ce JSON côté serveur, pas une entrée utilisateur brute). new Chart(...) appelle la librairie Chart.js pour dessiner un graphique en ligne dans le <canvas id="graphe">.

7.3 — _th.html : en-tête de colonne triable (template inclus/réutilisé)

1-13
{% spaceless %} <th> {% if tri == col %} {% if ordre == "asc" %} <a href="?...&tri={{ col }}&ordre=desc">{{ titre }} ↑</a> {% else %} <a href="?...&tri={{ col }}&ordre=asc">{{ titre }} ↓</a> {% endif %} {% else %} <a href="?...&tri={{ col }}&ordre=desc">{{ titre }}</a> {% endif %} </th> {% endspaceless %}
Ce petit fichier est inclus 5 fois dans liste.html (une fois par colonne) via {% include "capteurs/_th.html" with col="..." titre="..." %} — ça évite de copier-coller le même bloc HTML 5 fois. Logique : si la colonne actuelle (col) est celle déjà triée (tri == col), on affiche une flèche (↑ ou ↓) et un lien qui inverse le sens du tri au prochain clic. Sinon, on affiche juste le titre cliquable qui, au clic, trie cette colonne en desc par défaut. Tous les autres filtres actifs (f_capteur, f_debut...) sont systématiquement réinjectés dans le lien pour ne pas les perdre en changeant de tri.

7.4 — detail.html : édition et suppression d'un capteur

10-34
<form method="post"> {% csrf_token %} <input type="text" value="{{ capteur.id }}" disabled> <input type="text" name="nom" value="{{ capteur.nom }}" required> <button type="submit">Enregistrer</button> <button type="submit" name="supprimer" class="btn-danger" onclick="return confirm('Supprimer ce capteur ET toutes ses mesures ?')"> Supprimer </button> </form>
method="post" ici (contrairement à liste.html) car ce formulaire modifie des données côté serveur — convention HTTP : GET pour lire, POST pour écrire. {% csrf_token %} insère un jeton de sécurité obligatoire en Django pour tout formulaire POST : il protège contre les attaques CSRF (un site malveillant qui forgerait une fausse requête vers ce formulaire à l'insu de l'utilisateur) — sans ce tag, Django refuse la requête (erreur 403). Les champs disabled (ID, pièce) sont affichés mais non modifiables et donc non envoyés au serveur — leur protection est donc garantie côté navigateur ET côté serveur (puisque views.py ne lit jamais ces champs dans request.POST). Le bouton "Supprimer" a un attribut name="supprimer" distinct du bouton "Enregistrer" : c'est ce qui permet à views.py de savoir lequel des deux boutons a été cliqué via "supprimer" in request.POST. onclick="return confirm(...)" ajoute une boîte de dialogue JavaScript de confirmation avant même l'envoi du formulaire — si l'utilisateur clique "Annuler", confirm() renvoie false et le formulaire n'est jamais soumis.
